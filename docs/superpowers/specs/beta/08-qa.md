## Multi-user data isolation test plan

**Premise Engineering will push back on:** "All queries are parameterized; a contamination test is theatre." **Why I will not yield:** parameterization protects against SQL injection. It does **nothing** against a query that simply omits the `WHERE user_id = $X` clause. The mesocycles/programs/planned-sets routes were built under the single-user assumption (`PLACEHOLDER_USER_ID`) — the risk that one of these handlers returns User A's row to User B is real and unverified.

**Test pattern (every row below):** create users A and B in `beforeAll`; mint a token for A; perform an action on User B's resource using A's token; assert 404 (preferred) or 403, never 200 with B's data. Where a route lists resources, assert the list contains zero of B's rows.

### Contamination matrix

| Method | Route | Owns by | Required isolation test |
|---|---|---|---|
| GET | `/api/program-templates` | system | N/A — public catalog |
| GET | `/api/program-templates/:slug` | system | N/A |
| POST | `/api/program-templates/:slug/fork` | actor | Forked `user_program.user_id` = caller, never the body |
| GET | `/api/user-programs?include=` | actor | List MUST NOT include another user's rows; cross-user attempt returns only A's |
| GET | `/api/user-programs/:id` | actor | A reading B's program ID returns 404 |
| PATCH | `/api/user-programs/:id` | actor | A patching B's program returns 404, no row mutation |
| GET | `/api/user-programs/:id/warnings` | actor | A reading B's warnings returns 404 |
| POST | `/api/user-programs/:id/start` | actor | A starting B's program returns 404; partial-unique-index check is per-user (A active does not block B) |
| GET | `/api/mesocycles/today` | actor | Returns A's run only; B's active run is invisible to A |
| GET | `/api/mesocycles/:id` | actor | A reading B's run → 404 |
| GET | `/api/mesocycles/:id/volume-rollup` | actor | A reading B's rollup → 404 |

**Coverage target:** every row above has a passing integration test before Beta. **No exceptions for "obvious" routes** — the obvious ones are exactly the ones that get a missing-WHERE bug.

### Test infra requirement

A reusable fixture: `mkUserPair(): Promise<{a, b, tokenA, tokenB}>` in `api/tests/helpers/`. Most contamination tests are 5 lines: setup pair, perform A's action against B's resource, assert 404. Cheap to write, free to maintain.

## Performance + load plan

**Position I'm taking:** Beta admits a small cohort (≤25 users) on a single $X/mo Unraid box behind a single API process and a single Postgres. We are not load-testing for scale; we are load-testing to find **the cliff** — the concurrency at which `/api/mesocycles/today` falls over because the recap-stats CTE does a full table scan on a cold cache.

### Concurrency target for Beta

- **25 concurrent active users** as the design target. Each "user" = one human refreshing the today page, scrolling the chart, occasionally checking recap.
- **Realistic peak:** ~5 concurrent because users open the app once per workout. 25 is the worst-case "everyone trains at 6am" scenario.

### Latency budget (p95 from the LAN, post-Cloudflare)

| Endpoint | Hot/cold | p95 budget | Why this number |
|---|---|---|---|
| `GET /api/mesocycles/today` | hot | **< 200 ms** | Front-of-house. Renders before the user can blink. |
| `GET /api/health/weight?range=30d` | hot | **< 250 ms** | Chart endpoint, called on every load. |
| `GET /api/health/sync/status` | hot | **< 100 ms** | Tiny computed pill. Already `Cache-Control: max-age=60`. |
| `GET /api/user-programs?include=past` | warm | **< 400 ms** | My Library list page; loaded less often. |
| `GET /api/mesocycles/:id/volume-rollup` | warm | **< 500 ms** | Aggregation query; opened a few times per week. |
| `GET /api/mesocycles/:id/recap-stats` | cold | **< 800 ms** | End-of-meso; the per-exercise PR CTE is the heaviest read in the system. Block-comment in `mesocycles.ts` flags this. |
| `POST /api/user-programs/:id/start` | cold | **< 2000 ms** | Materialization. SERIALIZABLE + retry-on-40001. Once per mesocycle, OK to be slow. |
| `PATCH /api/planned-sets/:id` | hot | **< 150 ms** | Tap-tap-tap during a workout. Anything slower feels broken. |
| `POST /api/planned-sets/:id/substitute` | warm | **< 300 ms** | Mid-session swap. |
| `POST /api/health/weight` | hot | **< 200 ms** | iOS Shortcut bearer-token write. argon2 verify is the slow path; ensure it's bounded. |

### Test approach

1. **k6** scripts in `tests/perf/`. Why k6: scriptable, produces structured JSON for thresholding, no Node-loop interference.
2. **Two scenarios per endpoint:**
   - **Steady state** — 25 VUs for 2 min, ramping. Threshold: p95 below the budget above.
   - **Burst** — spike from 1 to 50 VUs in 5s, hold 30s. Threshold: zero 5xx, p99 below 2× budget.
3. **Run against the production container topology** on a sibling Unraid container or local docker-compose mirroring the s6-rc tree. Not against `npm run dev` — different process model, different numbers.
4. **Cold-cache discipline:** restart Postgres between scenarios when measuring "cold." The recap-stats CTE is the test that matters here.
5. **Output:** committed `tests/perf/beta-baseline-$(date).json` artifact and a one-page `tests/perf/README.md` with the cliff identified per endpoint.

### What this catches that unit tests miss

- N+1 queries. Today's contract tests use a freshly-forked program with one set; they will not surface a "loops over `day_workouts` and queries `planned_sets` per day" mistake the way 25 VUs against a 5-week mesocycle will.
- argon2 cost. Token verification on every request multiplied by 25 concurrent users — is the configured cost-factor sustainable?
- `pg.Pool` saturation. The pool is sized in `api/src/db/client.ts`; we have no measurement that 25 concurrent users don't exhaust it.

### Engineering will push back

- **"Beta is small, we don't need load testing."** Counter: if we don't measure now, we will not know when we should worry. The cost is one engineer-day of k6 scripts. The cost of finding the cliff via a real-user incident is the user.
- **"The recap CTE is fine, I optimized it."** Counter: prove it. Cold-cache p95 number or it didn't happen.

## Data-loss risk assessment

### Current state (alpha)

- DB content is throwaway per memory `project_alpha_state.md`.
- Single user. `c5a79f4b-…` has one in-flight mesocycle (`eca01e1a-…`) per the latest handoff.
- No backup automation; relies on Unraid host snapshots which have not been tested for restore-into-a-running-RepOS-container.

### Beta cutover decision (force Engineering to pin this down)

**Two paths. Pick one explicitly. Document the choice.**

**Path A — Hard reset at Beta cutover.** Wipe the alpha DB, re-create empty, Beta users start with no history. Simplest. Eliminates the "did we corrupt the user's first mesocycle" risk. The user (jmeyer@) loses their alpha mesocycle data; per `project_alpha_state.md`, that's acceptable.

**Path B — Migrate alpha → Beta.** Run a forward migration that takes the existing data as the new baseline. Requires:
- A documented data audit before cutover (row counts per table, check constraints all green, no orphan rows).
- A taken-and-verified pre-cutover dump (per B5).
- A documented rollback if migration corrupts data.
- A second dry-run on a clone of production data.

**My recommendation:** Path A. The alpha data is one user's training history; the cost of preserving it is dwarfed by the migration risk. **Hold the line:** if Engineering picks Path B, B5 + B6 become non-negotiable AND the migration must be dry-run on a clone first. No "let's just see if it works" against the real DB.

### Post-Beta risks (regardless of path)

- **Single Postgres, no replication.** Container loss = data loss back to the most recent backup. Backup cadence MUST be specified (hourly? nightly? what's the RPO?).
- **No off-box backup verified.** "Unraid host snapshots solve it" — has anyone restored from one? If no, the assertion is unproven.
- **No PITR (point-in-time recovery).** A bad migration at noon means rollback to last-night's backup. Acceptable for Beta? Engineering call, but document it.
- **Materialize-at-start means hot-patches don't reach active runs** (per sports-med spec). If we discover a clinical issue with a curated program post-launch, users on active runs keep the buggy plan. This is a *programming* hazard, not strictly data loss, but it's adjacent.

## Bug-triage + rollback plan

### Severity tiers + time-to-mitigate

| Tier | Definition | Time-to-mitigate | Action |
|---|---|---|---|
| **Sev-1** | Data corruption, data leak across users, full outage, auth bypass. | **30 min to mitigate, 4h to fix.** | Page on-call (jason@). Roll back container to previous `:sha-XXXXXXX` image. Take Postgres snapshot before rollback. Disclose to affected users within 24h. |
| **Sev-2** | Critical user flow broken (cannot start meso, cannot log workout, cannot post weight). Users blocked but data safe. | **2h to mitigate, 24h to fix.** | Roll back image. If migration-induced, decision tree: (a) backward-compat? roll forward-fix; (b) not backward-compat? roll back DB to pre-migration snapshot. |
| **Sev-3** | Visual regression, slow but functional, recovery flag misfires, term tooltip wrong. | **1 business day.** | Fix on next deploy cycle. |
| **Sev-4** | Cosmetic, edge-case wording, copy nits. | **Next regular release.** | Backlog. |

### Rollback procedures (required runbook entries)

1. **Container-image rollback** — automate the recipe from memory `reference_unraid_redeploy.md`. New script `docker/scripts/rollback.sh <sha>`:
   ```
   ssh unraid "docker stop RepOS && docker rm RepOS && \
     docker run -d --name RepOS --net=br0 --ip=192.168.88.65 \
       --env-file /mnt/user/appdata/repos/.env \
       -v /mnt/user/appdata/repos:/config \
       ghcr.io/otahesh/repos:sha-<sha>"
   ```
   Test the script. The user has burned 30 min on `docker restart` not picking up `:latest` once already; never again.

2. **DB snapshot before any deploy** — pre-deploy hook in CI: SSH to Unraid, run `pg_dump` inside the container, drop to `/mnt/user/appdata/repos/snapshots/pre-deploy-<sha>-<ts>.sql.gz`, retain last 10. (See B5.)

3. **DB-state rollback** — documented procedure: stop container, restore latest pre-deploy snapshot, restart container with the previous image SHA. Tested as part of B6.

4. **Migration rollback** — every migration ships with an inverse. CI rejects a migration PR that does not include both directions. (Or pick a strict "forward-only with new migrations to undo" policy and document it. **Pick one.** Don't leave it ambiguous.)

5. **External smoke after every deploy** (B13) catches the "container is running but broken" failure mode that wouldn't otherwise show.

### Dry-fire requirement (B10)

Before opening Beta: pick a Tuesday, schedule 30 min, simulate a Sev-1. Engineering rolls back. Does it work? Time it. If >10 min from declaration to mitigation, the runbook is wrong and gets fixed before Beta.

## Pre-Beta security review checklist

The monolithic-container deploy closed seven MUST-FIX items pre-public-launch. **Beta scope adds new attack surface** (programs, mesocycles, planned-sets, recovery-flags, the equipment profile, the recap-stats endpoint) that did not exist when that review happened. A **focused re-review** is required.

### Authentication / authorization

- [ ] Every new route registered with `requireBearerOrCfAccess` preHandler. Audit by grep — any route with `app.get(` / `app.post(` etc. that lacks an auth preHandler is a finding.
- [ ] Every handler that touches user data uses `req.userId` (set by middleware), NEVER reads `user_id` from request body or querystring as the identity source.
- [ ] CF Access path: body `user_id` IGNORED. Admin-key path: body `user_id` honored. The auth-mode confusion test in §"Multi-user data isolation" enforces this.
- [ ] Token scopes enforced for write operations. `program:write` scope check exists for any planned-set / customize / substitute / start endpoint? Audit `hasScope(...)` call sites.
- [ ] Token verification (argon2) on every request — confirmed bounded latency under load (B9).
- [ ] Revoked tokens reject within 60s of revocation (no over-cached identity).
- [ ] Expired CF Access JWT returns 401 cleanly, not 500 — frontend recovery path tested (B3).

### Multi-user / IDOR

- [ ] Every contamination test in §"Multi-user data isolation" passes. **The deepest path** (`PATCH /api/planned-sets/:id` reaches user via day_workout → mesocycle_run → user_id, three joins) is the highest-risk lookup; eyeball the SQL personally during review.
- [ ] No route returns a list without an explicit `WHERE user_id = $1` clause except documented system catalogs (`/api/exercises`, `/api/program-templates`, `/api/muscles`).

### `PLACEHOLDER_USER_ID` migration

- [ ] All references in `frontend/src/auth.tsx` removed; replaced with real CF-Access-derived identity.
- [ ] Tests that referenced `PLACEHOLDER_USER_ID` updated to use a per-test fixture user.
- [ ] No path can result in a row being written with `user_id = '00000000-0000-0000-0000-000000000001'` in production (assert via grep + a runtime guard that rejects this UUID at insert-time).

### Rate limits + abuse

- [ ] Per-IP rate limit at nginx is in place (per monolithic-container plan); confirm it covers `/api/*` and is appropriate for a 25-user Beta (not so tight it blocks normal use, not so loose it allows credential stuffing).
- [ ] `POST /api/health/weight` retains its 5-writes-per-day-per-user 409 limit.
- [ ] `POST /api/tokens` limited per IP (admin-key + CF Access already gates it; double-check nginx limit covers).
- [ ] No expensive endpoint (recap-stats, materialize) is open to anonymous bursts.

### Input validation

- [ ] Every route validates body via Zod schema (per program-model plan migration). Audit complete.
- [ ] Calendar-valid date check on weight (already in scope per monolithic-container plan).
- [ ] UUID validation on all `:id` path params (Zod's v4 `.uuid()` semantics — version digit 1-8 or nil/max — confirmed correct per session-handoff fix to plannedSets contract test).
- [ ] No string length is unbounded on any text input that lands in DB (program name, set notes, override reason, label).

### Secrets / logging

- [ ] `req.headers.authorization` redacted in logs (already done per monolithic-container plan); audit that no new route logs the full request.
- [ ] No secret value (token plaintext, argon2 hash, DB password) appears in any error response body or log line.
- [ ] `.env` not in any commit; pre-commit hook would help.

### Frontend

- [ ] No bearer token in localStorage. Tokens live in HttpOnly cookie OR are not browser-side at all (CF Access JWT is the browser-side identity; bearer is only iOS-Shortcut).
- [ ] No XSS surface from user-supplied program names / overrides / set notes — React escaping confirmed, `dangerouslySetInnerHTML` not used.
- [ ] CSP set by nginx is restrictive; new bundle changes don't loosen it.

### Closure rule (per `feedback_ship_clean.md`)

Critical AND Important findings closed before Beta. No "v1.5 backlog" deferrals. If a finding cannot be closed, it gets explicit accept-residual-risk sign-off from the engineering lead, written down, with a date.

## Beta feedback loop spec

A Beta finding that dies in a user's head is an unfixed bug.

### Required components

1. **In-app "Send feedback" affordance.** Bottom-of-AppShell or settings page. Single textbox, optional screenshot capture, optional contact-back-with checkbox. POSTs to `/api/feedback`.
2. **`/api/feedback` route.** Captures: text, user_id, current URL (so we know what surface they were on), user-agent, app build SHA, timestamp. Writes to a `feedback` table. Optionally also forwards to email/Slack via webhook.
3. **Feedback table** with `id`, `user_id`, `body`, `route`, `app_sha`, `created_at`, `triaged_at`, `triaged_by`, `status (open|triaged|closed|wontfix)`.
4. **Triage cadence.** Engineering reviews `feedback` rows daily during Beta. Sev classification per §"Bug-triage." Loop closes when row is `closed` AND user has been replied-to (manually for now).
5. **External validation** — actually submit a test feedback as a non-admin user. Per memory `feedback_verify_external_config.md`, "configured" ≠ "working." Curl-equivalent the post and verify the row lands.

### Beta cohort sizing

- **Recommended cap:** **5–10 users** for first 2 weeks. 25 thereafter if green. Engineering will likely propose larger ("we want feedback") — **hold the line at 10 for week 1**. If something is broken, ten people see it and it's fixable; fifty people see it and trust collapses.
- Each Beta user knows they're a Beta user (explicit in the disclaimer / first-run flow per sports-med #1). Sets expectation.
- Each Beta user has a documented contact path back to the engineer (the `/api/feedback` loop above is the system-of-record; email is the human path).

### Anti-pattern to refuse

"We'll add the feedback loop after Beta starts." No — a feedback loop that arrives in week 3 of Beta means weeks 1–2 of bugs are oral history. Wire it before cutover.

## Things I expect Engineering to try to defer (and why I'll resist)

Each of these is something I expect to hear in Round 2. Each is followed by what I'll say back.

- **"Multi-user contamination tests are theatre — parameterized SQL prevents this."**
  *Risk:* Parameterization stops injection, not missing-WHERE clauses. The frontend was built with `PLACEHOLDER_USER_ID` for months. Confidence that every handler uses `req.userId` AND filters by it is unverified. **Hold the line: every row in the matrix gets a test.**

- **"E2E browser tests are slow; we'll add them in v1.5."**
  *Risk:* Auth flow + golden-user-journey are the highest-leverage tests. They catch the bugs that unit tests by design cannot. Playwright MCP is already available. **Hold the line: 15 E2E tests before Beta admission.**

- **"Performance is fine, we don't need load tests for 5 Beta users."**
  *Risk:* Beta will scale to 25. The recap-stats CTE has not been measured under load. Discovering the cliff in production = the user is the cliff-finder. **Hold the line: k6 baseline before Beta.**

- **"Backup is solved by Unraid host snapshots."**
  *Risk:* Untested restore = no backup. The user does not actually know that an Unraid snapshot of `/mnt/user/appdata/repos/` produces a usable Postgres state (running Postgres = inconsistent on-disk state without a `pg_dump`). **Hold the line: tested `pg_dump` + restore-into-clone procedure.**

- **"Migration rollback isn't necessary; we roll forward."**
  *Risk:* "Roll forward" for a migration that just corrupted data means the corruption is permanent. **Hold the line: every migration ships with an inverse, OR explicit accept-risk sign-off per migration with a documented data-rescue plan.**

- **"The Important security findings can wait for v1.5."**
  *Risk:* `feedback_ship_clean.md` says no. Important is not Nice-to-have. **Hold the line.**

- **"User-reachability audit is implicit — if it's coded, the user can find it."**
  *Risk:* `feedback_user_reachability_dod.md` exists because this is a recurring failure pattern. The most recent one (recap + swap) is in-flight today. **Hold the line: explicit audit, walked through, in writing.**

- **"PR-blocking on CI test coverage adds friction."**
  *Risk:* Beta starts with red tests landing on `main`. **Hold the line: required checks include all three test jobs.**

- **"We'll add `/api/feedback` after we see the first real Beta complaint."**
  *Risk:* Weeks 1–2 of Beta findings are lost to the floor. **Hold the line: feedback loop wired before cutover.**

- **"The `PLACEHOLDER_USER_ID` only exists in the frontend, it's harmless."**
  *Risk:* It will leak. The first time a real user has zero rows, a code path may helpfully insert with the placeholder. **Hold the line: grep-block the constant, runtime-reject the UUID at DB layer.**

## Risks / unknowns

- **Auth scheme for the web frontend at Beta.** The plan implies CF Access JWT becomes the browser-side identity. Confirm with Engineering — is there an in-app login? CF Access redirect-only? What happens on token expiry mid-workout? (B3 covers the test, but the *design* needs to be explicit.)
- **Live-Logger (set_logs) timing.** Sports-med spec hinges on `set_logs` being written by Beta. If Live Logger slips, recovery flags ship dead. QA cannot test what doesn't exist. **Verify Engineering's commitment to Live Logger as Beta-blocking.**
- **No staging environment.** Today, `repos.jpmtech.com` is production, period. Where do load tests run? Where do Playwright tests run? `docker-compose` locally is fine for dev, but Beta needs a non-prod environment to validate the full stack with real DNS, real CF Tunnel, real Access policy. Either spec a staging container at a sibling IP/hostname OR formalize "we test against prod with a feature flag" — but neither is in scope today and that's a gap.
- **No alerting.** The container could be down for hours and we'd find out from a user. Beta needs at least: external uptime monitor (UptimeRobot or sibling) pinging `repos.jpmtech.com`, and a Postgres-disk-full alarm. Not in any spec I've seen.
- **Test isolation within api/tests.** Many tests share the `repos_test` DB and rely on `--no-file-parallelism` (per `package.json`). Adding ~50 integration tests will lengthen the suite linearly. Plan: budget the runtime, consider per-file DB transactions with rollback in afterEach, OR ephemeral PG per CI run.
- **iOS Shortcut bearer-token rotation.** When a Beta user rotates a token, the Shortcut on their phone breaks silently until re-pasted. UX path for rotation (in-app + Shortcut update guide) needs spec. Out of QA scope but flagging.
- **Time-zone correctness at the DB layer.** Spec stores wall-clock `time` as display label only; user TZ on user record. Test that mesocycle/today crossing midnight in a non-UTC TZ behaves correctly. Already covered by `userLocalDate.test.ts` at the unit level — needs an integration test crossing a real midnight.
